# Phase 2 — UI Design Contract

> Visual and interaction contract for the People and Items input surface. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (native HTML + Tailwind utility classes) |
| Icon library | none (no icons required in this phase) |
| Font | system-ui, 'Segoe UI', Roboto, sans-serif |

**Rationale:** Research explicitly ruled out shadcn, Radix, and Headless UI. All components in this phase (text inputs, buttons, chip toggles) are achievable with Tailwind utility classes on native HTML elements. No modals, dropdowns, or complex widgets exist in this phase.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding within chips |
| sm | 8px | Gap between chips, compact element margins |
| md | 16px | Default element spacing, panel internal padding, item list gap |
| lg | 24px | Section padding (PeoplePanel, ItemsPanel) |
| xl | 32px | Gap between major sections (people panel to items panel) |
| 2xl | 48px | Not used in this phase |
| 3xl | 64px | Not used in this phase |

Exceptions: Chip touch targets use `min-h-[44px] min-w-[44px]` per WCAG 2.5.5. This is not a spacing token but a minimum interactive size constraint.

**Usage in this phase:**
- **Display (28px/700):** App title "Split the Bill" in the header
- **Heading (20px/700):** Section headings "People" and "Items"
- **Body (16px/400):** Person names, item descriptions, price values, input text

**Font stack:** `system-ui, 'Segoe UI', Roboto, sans-serif` -- inherited from existing project, no web font download needed.

## Color

| Role | Light Value | Dark Value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) | `#ffffff` (white) | `#111827` (gray-900) | Page background, main surfaces |
| Secondary (30%) | `#f9fafb` (gray-50) | `#1f2937` (gray-800) | Panel backgrounds, card surfaces, input backgrounds |
| Accent (10%) | `#3b82f6` (blue-500) | `#3b82f6` (blue-500) | Active assignment chips only, primary "Add" buttons |
| Destructive | `#ef4444` (red-500) | `#ef4444` (red-500) | Delete/remove buttons only |
| Warning | `#f59e0b` (amber-500) | `#f59e0b` (amber-500) | Unassigned item border and label |

**Accent reserved for:**
1. Assignment chip background when a person IS assigned to an item (`bg-blue-500 text-white`)
2. "Add Person" button
3. "Add Item" button

**Warning reserved for:**
1. Unassigned item card border (`border-amber-400`)
2. Unassigned item card background tint (`bg-amber-50` / `dark:bg-amber-900/20`)
3. "Not assigned to anyone" label text (`text-amber-600`)

**Neutral palette:**
- Borders: `#e5e7eb` (gray-200) light / `#374151` (gray-700) dark
- Muted text: `#6b7280` (gray-500) light / `#9ca3af` (gray-400) dark
- Inactive chip background: `#f3f4f6` (gray-100) light / `#374151` (gray-700) dark
- Inactive chip text: `#4b5563` (gray-600) light / `#d1d5db` (gray-300) dark

**Dark mode:** Automatic via `prefers-color-scheme` media query. Tailwind v4 `dark:` variant uses this by default. No manual toggle in this phase (deferred to Phase 4).

## Component Visual Contracts

### App Shell
- Full viewport height: `min-h-screen`
- Max content width: `max-w-lg` (32rem / 512px), centered with `mx-auto`
- Header: centered text, bottom border, `p-4`
- Main content: `p-4 space-y-6` (24px gap between panels)

### PeoplePanel
- Section heading: "People" at heading size (20px/700)
- Add form: single text input + "Add Person" button, horizontal layout on one line
- Input placeholder: "Enter name"
- Button style: accent background (`bg-blue-500 text-white`), rounded (`rounded-lg`), `px-4 py-2`
- Person list: vertical stack with `space-y-2` (8px gaps)

### PersonRow
- Layout: horizontal flex, person name on left, delete button on right
- Name display: body text (16px/400), clickable for inline edit
- Inline edit: on click/tap, text becomes an input field. Save on blur or Enter. Cancel on Escape.
- Edit input: bordered (`border border-gray-300`), rounded (`rounded`), `px-2 py-1`
- Delete button: text "Remove Person", destructive color (`text-red-500`), minimum 44px touch target
- Row padding: `py-2 px-3`

### ItemsPanel
- Section heading: "Items" at heading size (20px/700)
- Add form: two inputs (description + price) + "Add Item" button
- Description input placeholder: "Item name"
- Price input placeholder: "$0.00"
- Price input: `inputMode="decimal"` for mobile numeric keyboard
- Button style: same as PeoplePanel add button
- Item list: vertical stack with `space-y-4` (16px gaps, provides breathing room for chip rows within each item)
- Empty state (no items): "No items yet. Add receipt items above."
- Empty state (no people exist): Items panel shows hint below assignment area: "Add people first to assign items"

### ItemRow
- Card style: bordered (`border border-gray-200 rounded-lg p-3`)
- Unassigned state: `border-amber-400 bg-amber-50 dark:bg-amber-900/20` with warning label
- Layout: item name and price on first line (flex, space-between), assignment chips on second line
- Item name: body text (16px/400), clickable for inline edit (same pattern as PersonRow)
- Price: body text (16px/400), right-aligned, formatted as "$X.XX"
- Price inline edit: same input pattern, `inputMode="decimal"`
- Delete button: same destructive style as PersonRow delete
- Split mode toggle: text button or small toggle between "Shared" and "Assigned" labels

### AssignmentChips
- Container: `flex flex-wrap gap-1.5` (6px gap between chips)
- Chip size: `min-h-[44px] min-w-[44px] px-3 py-1.5` (44px minimum touch target)
- Chip shape: `rounded-full`
- Chip text: label size (14px/400)
- Active chip: `bg-blue-500 text-white`
- Inactive chip: `bg-gray-100 text-gray-600 dark:bg-gray-700 dark:text-gray-300`
- Transition: `transition-colors` for smooth toggle feedback
- Hidden when: no people exist in the store

**Destructive actions in this phase:**
1. **Remove Person** -- immediate, no confirmation dialog. Rationale: low data loss (just a name), and the store already cleans up item assignments automatically.
2. **Remove item** -- immediate, no confirmation dialog. Rationale: quick re-entry is easy; modal confirmations slow down restaurant-table usage.

## Interaction Contracts

### Add Person Flow
1. User types name in text input
2. User taps "Add Person" button or presses Enter
3. Input clears, new person appears in list below
4. Focus returns to input for rapid sequential entry

### Add Item Flow
1. User types item name in first input
2. User types price in second input (dollar format, e.g. "12.50")
3. User taps "Add Item" button or presses Enter
4. Price is parsed to integer cents on submission
5. Input fields clear, new item appears in list below
6. If price parse fails, show inline error: "Enter a valid price (e.g. 12.50)"

### Inline Edit Flow
1. User taps on a person name or item name/price
2. Text transforms into a focused input with current value
3. User edits the value
4. Save: blur or Enter key commits the edit
5. Cancel: Escape key reverts to original value

### Assignment Chip Toggle Flow
1. Each item row displays one chip per person in the store
2. Tapping a chip toggles that person on/off for the item
3. Active chips show accent color (blue-500), inactive show gray
4. When toggling sets `assignedTo` to non-empty, `splitMode` is set to 'assigned'
5. Split mode toggle ("Shared" / "Assigned") controls how the engine interprets the selection

### Unassigned Item Flagging
1. Any item with `splitMode === 'assigned'` and `assignedTo.length === 0` is visually flagged
2. Flag consists of amber border + background tint + "Not assigned to anyone" label
3. Flag disappears immediately when at least one person is assigned

## Accessibility Notes

- All inputs have associated labels (visible or via `aria-label`)
- Assignment chips use `<button>` elements (not `<div>`) for keyboard accessibility
- Active chips convey state via `aria-pressed="true"` in addition to color
- Minimum touch target: 44x44px on all interactive elements
- Price inputs use `inputMode="decimal"` for appropriate mobile keyboard
- Inline edit fields receive `autoFocus` when activated
- Color is never the sole indicator of state (unassigned items have both color AND text label)
